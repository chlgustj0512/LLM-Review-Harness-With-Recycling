## 0. 통합 결정 사항 (충돌 해소 + 핵심 가치)

세 문서를 이어 붙일 때 두 군데서 실질 충돌이 났고, 사용자 결정으로 다음과 같이 해소했다.

### 0-1. 충돌① 점수 체계 → **레이어 분리**
- **문제:** [A]는 "절대 눈금(threshold)을 폐기하고 *이전 챔피언 대비 상대 변화*로 순위를 매긴다"(동률밴드 ε도 폐기). [B]는 "0~100 **절대 점수** + 단계별 threshold(60/75/85…)"를 쓴다. 정면 충돌.
- **결정:** 두 장치는 **서로 다른 질문에 답한다.** 충돌이 아니라 분업으로 처리한다.
  ```
  절대 점수·Hard Gate (문서 B)  →  "이 후보가 통과 자격이 되나?"   (통과/탈락 = 게이트)
  상대 사전식 래칫 (문서 A)      →  "통과한 것들 중 누가 챔피언인가?" (순위 = 래칫)
  ```
  즉 **게이트는 절대 컷, 순위는 상대 래칫.** [A] §게이트+래칫도 원래 "1단계 게이트(통과/탈락) + 2단계 래칫(순위)"의 2단 구조라 이 분리와 정합한다.

### 0-2. 충돌② 흐름 구조 → **결합(순차 로드, 빠짐없이)**
- **문제:** [B]는 "3단계 순차 게이트"(정→반 듀오를 생존→정제→승급 3회 반복). [A]는 "다측면 병렬 자가선별"(사전 도메인 라우팅 폐기).
- **결정:** 서로 다른 축이라 **결합한다.**
  ```
  3단계(생존→정제→승급) = 시간축 큰 흐름        (문서 B)
  각 단계 내부 심사       = 다측면 자가선별        (문서 A)
  ```
- **실행 모드(사용자 확정):**
  - **순차 로드.** 다측면 심사관 모델을 한 번에 하나씩 메모리에 올렸다 내린다(개발기 32GB RAM 제약). 병렬 동시성을 추구하지 않는다.
  - **빠짐없이 전수 통과.** "빨리 솎아내는" cheap-first 사전 필터·차등 심사 깊이·통과율 측정 장치를 **두지 않는다.** 모든 후보가 모든 관할 심사관을 거친다.
  - **자가선별의 역할 한정.** 자가선별("내 관할?/아님 빠짐")은 *비용 절감*이 아니라 *오라우팅 제거 + 관할 밖 발언 배제*라는 **정확성** 장치로만 유지한다.

### 0-3. 통합 형태 → **단일 완전 통합본, [A]가 메인**
부록 분리 없이 하나로 녹였다. 골격은 [A], 검증 규격은 [B], 적대 진화는 [C].

### 0-4. ⚠️ 핵심 가치 재확인 (이 시스템이 빠르지 않은 이유)
> **이 엔진이 원하는 것은 "빠른 합·불합 구별"이 아니라 "실패조차 재활용하는 것"이다.**
> 느리게 가도 빠짐없이 처리하는 쪽이 낫다. 후보를 빨리 버리면 라이브러리(§11)와 부정예시(§12)로 환원할 *재료 자체를 버리는 것*이기 때문이다. 속도·비용 최적화는 이 시스템의 1차 목적이 아니며, 정확성·재활용성과 충돌하면 후순위다.

### 0-5. 재심(Appeal) → **외부 Claude 문서 경유 (자동 API 아님)**
- **결정:** 재심·이종 계통 검증을 시스템이 *자동 API 호출*로 처리하지 않는다. 대신 재심이 필요한 케이스를 **재심 요청 문서(Appeal Request Packet)로 출력**하고, 사용자가 그 문서를 구독 중인 Claude 채팅에 붙여넣어 검증받은 뒤, 결과를 시스템에 수동 재입력한다.
- **근거:** §1 자문 ②("사람이 영점인가") 원칙과 정확히 부합. 재심 = *사람 + 외부 이종 계통(Claude)* 이 함께 영점을 잡는 단계. Claude API 키 없이 구독 채팅만으로 "오픈웨이트 공통 맹점을 끊는 그물"(약점 대응)을 확보한다.
- **상세:** 트리거·문서 규격·재입력 절차는 §13-1, 복붙용 템플릿은 **부록 C**.

### 0-6. 구성원 모델 7+1명 확정 (Lunar Lake 32GB 순차 로드)
- **결정:** 로컬 심사단을 **계통(lineage)을 의도적으로 분산한 7명**으로 구성하고, 8번째 재심관은 외부 Claude(문서 경유)가 맡는다. 명단·이유·크기·계통 거리는 §7-1.
- **선정 원칙:** "많이"가 아니라 "멀리". 선정 기준은 성능 순위가 아니라 **계통 거리**(누가·어떤 데이터로 학습시켰나). 같은 회사·distillation 관계는 *맹점이 겹쳐* 다측면이 허울만 남는다.
  > ⚠️ **결정적 함정:** `deepseek-r1`의 로컬 소형 태그(1.5b~14b)는 독립 모델이 아니라 **Qwen 2.5를 미세조정한 distillation**이다. Qwen 심사관 옆에 두면 계통 독립 효과 ≈ 0. 구성원에서 제외하거나, 쓰려면 "Qwen과 같은 계통"으로 취급한다.
- **Lunar Lake 적합성(⚠️ 일부 실측·일부 추정):** 이 칩 iGPU는 *프롬프트 읽기(prefill)는 빠르고 토큰 생성은 느리다.* 심사관은 "긴 후보를 읽고 짧은 판정을 내는" 작업이라 이 강점과 맞고, 생성기만 MoE(활성 파라미터 작음)로 빼면 생성 속도 부담도 준다. iGPU 가속은 **IPEX-LLM 백엔드**로 가능하나 설정이 까다롭다(§7-1 주).

# Part I — 설계 원칙과 핵심 명제 (출처 [A])

## 1. 설계 원칙 — 이 시스템이 장치가 적은 이유

이 시스템의 엔진은 하나다: **사전 보증을 사후 도태로 바꾼다.** 미래의 모든 잘못을 막는 장치를 *앞에* 세우는 대신, 틀린 것이 *나온 뒤* 거르고 되돌린다. 인간의 유추·직관이 작동하는 방식이며("틀린 유추"의 존재가 곧 사전검증을 안 한다는 증거), OR 게이트(§5)와 진화적 도태의 일관된 귀결이다.

**요소를 "미해결/방어 필요"로 올리기 전 3개 자문 — 하나라도 Yes면 올리지 않는다:**

1. **사후로 되나?** 사전 보증 대신 "틀리면 그때 거르고 라이브러리 환원"으로 풀리나?
2. **사람이 영점인가?** 사용자가 통보·검토하면 되나? (영점=절대 기준은 사람·도구가 잡고, LLM은 눈금만.)
3. **상대계로 충분한가?** 절대 눈금(ε·임계값·기준선) 없이 "이전 대비 변화"로 풀리나?

**추가 축 — 진리 기준이 아니라 유용성 기준:** 산출물은 *참이라서*가 아니라 *쓸모 있어서* 통과한다(뉴턴역학: 틀렸지만 200년 쓰임 → 나중에 검증이 와도 폐기가 아니라 *범위 재조정*). 검증은 통과의 전제가 아니라 *사후 정제(리팩토링) 도구*다. **단 예외:** 종속핵심(논리·정확성, §9)에서는 검증이 *필요조건*이다 — 거기선 게이트로 막는다.

> **비유:** 이 시스템은 사전(辭典)이다. 단어를 *어원이 옳아서*가 아니라 *사람들이 써서* 싣고, 나중에 어원 연구가 오면 "주의: 민간어원"을 *덧붙이지*(리팩토링) 폐기하지 않는다.

## 2. 핵심 명제 — OR 게이트와 누설률

시스템의 하중을 받는 부품은 "변증"이라는 왕복 의식이 아니라 **독립적이고 다측면적인 심사 필터**다. 생성기(정)는 강하지만 *자족 불가*(자기 결함을 자가검출 못 함, §3). 부가가치는 정이 아니라 **반(심사)·재심·접지된 도구**에서 나온다.

이 시스템은 **OR 게이트**다: 크기가 뒤죽박죽인 날것의 후보를 거대한 필터에 던져 *통과하는 것 하나*를 원한다. 입구 품질을 정교하게 관리하는 AND 게이트가 아니다. 그래서 단 하나의 핵심 지표가 생사를 가른다 — **필터 누설률(통과한 것 중 결함 비율).**

> **용어:**
> - **OR 게이트** = 통과 1개면 이득. 입구 품질·신규성·다양성은 측정 안 함(되먹임을 안 하므로 측정 이유 없음; 못 쓰는 후보도 생성기 데이터셋엔 쌓임).
> - **false positive(거짓 통과)** = 떨어졌어야 할 결함물이 출고됨. OR 게이트의 최대 위험. *놓친 좋은 후보*는 OR 구조상 덜 치명적.
> - **비유:** OR 게이트는 "광맥에서 금 한 톨이라도 건지면 성공". 흙을 같이 퍼 올리는 건(나쁜 후보) 큰 손해가 아니지만, *가짜 금을 진짜로 통과시키는 것*(false positive)이 치명적.

## 3. 근거 — 다회차 반증 기록

핵심 결정은 사변이 아니라 *실제로 깨져 본* 결과다.

1. **코드 실험(best-of-N vs 루프):** visible test가 버그를 노출할 때만 루프 발화, 아니면 best-of-N으로 퇴화(3회 재확인). VADCA ablation 기여 = 제로. → 루프는 현재까지 best-of-N을 못 이김.
2. **철학 실험(반증):** 최강 생성기가 *자기가 차단 선언한* 오류(선결문제 요구 — 회귀논증을 한쪽에만 비대칭 적용)를 저지르고 자가검출 실패. 학부생 1명이 30초에 발견. → 생성기 자족 불가, 독립 반(反) 필요.
3. **분류 정정:** 그 결함은 "이음새"가 아니라 단일 측면(논리구조) 내부 결함(전제가 틀리면 논증이 틀림). 이 정정이 "이음새 심사관 상설 배치"를 필수→조건부로 강등.

**단일 법칙:** 루프 가치 = 검증기 피드백 풍부함의 함수. 얇은 pass/fail → 루프 ≈ best-of-N. 풍부·국소 피드백 → 루프가 이길 *기회*(필요조건, 충분조건 아님).

# Part II — 확정 아키텍처 (출처 [A])

## 4. 反/루프 분해

| 부품 | 상태 | 처리 |
|---|---|---|
| ① 反 (독립 검증) | **확정·필수** | 핵심. 전 도메인 적용 |
| ② 반복 수정 루프 | **미증명·보류** | 예산매칭 D′ 비교 통과 시에만 채택 |
| ③ salvage 메모리 | 별개·미검증 | A/B/C 실험이 답함 |
| ④ 재심(appeal) | **확정·문서 경유** | 외부 Claude(이종 계통)가 1차 통과/탈락을 재검. 자동 API 아님 → **재심 요청 문서 출력**(§13-1) → 사용자 수동 검증·재입력 |

> **용어:** **best-of-N** = 후보 N개 독립 생성 후 검증 통과 최고 1개 선택. **루프** = 1개→반이 깜→피드백으로 같은 후보 수리→반복.

## 5. 필터 철학 — OR 게이트와 누설률

- **OR 게이트:** 통과 1개면 이득. 입구 품질·신규성·다양성은 측정 안 함(못 쓰는 후보도 생성기 데이터셋엔 쌓임).
- 단 하나의 핵심 계기판 = **필터 누설률**(false positive 비율). 입구가 더러울수록 필터로 오는 결함 절대량이 늘어 false positive가 곱셈으로 커지므로, 입구를 비울수록 필터를 더 두들겨야 한다.

## 6. 발견(Detection) vs 수리(Repair)

- **A 발견:** 단서 없이 무엇이 틀렸는지 처음 짚음. 어려움.
- **B 수리:** 짚어주면 이해·수정. 쉬움.
- 생성기는 B 강함 증명(§3-2 즉시 납득·보강), **A는 미테스트**. 검증기(반)에 필요한 건 A. **B 점수로 A 자리에 앉히지 말 것.**
- **결정:** LLM 반을 쓰되 *발견율(A)을 도메인별 실측 전엔 단독 신뢰 금지.* 측정 = 알려진 결함 미끼(known-flaw probe) 블라인드 투입. 임계 이상이면 단독 승인, 미만이면 사람/도구 접지 강제.

## 7. 다측면 자가선별 심사 (+ 순차 로드)

```
정          : 강모델 1개 (생성 + 수리 B)
측면 심사관  : 다계통, 각자 1개 측면
자가선별     : 전 심사관에 후보를 다 던짐 → 각자 "내 영역?/아님 빠짐"
영역 게이트  : 심사관의 영역 밖 발언은 판정서에서 제외
```

- **사전 도메인 라우팅을 폐기한다.** "이 과제는 철학"이라고 *먼저 찍는* 단계를 없앰 → *오라우팅이라는 범주가 사라짐*(틀릴 사전판정이 없으니). 분류라는 별도 단계를 심사에 흡수.
- **다계통(cross-lineage) 필수:** 상관된 맹점 분산. **잔여 위험:** "다계통=독립"이 아니라 *부분상관*(학습 코퍼스 겹침) → §6 발견율 실측 없으면 안심 금지.
- **자가선별의 유일한 구멍 = 빈 측면:** 아무 심사관도 안 집은 측면은 무심사 통과(예: 윤리적 함의). → **빈 측면 탐지**: 안 집힌 측면이 있으면 *반려가 아니라 사용자 검토 호출 신호*. (분류기 아님, 빈칸 세기.) 현재 규모에선 사용자가 영점. ⚠️[확장 시: 사람 병목 → 재검토.]

### 7-1. 실행 모드 — 순차 로드, 빠짐없이 + 구성원 명단 (통합 결정 §0-2·§0-6)

**(1) 로딩 방식**
- 심사관 모델을 **한 번에 하나씩** 메모리에 로드해 모든 후보를 통과시킨 뒤 내리고, 다음 심사관을 로드한다. 통합 LPDDR5X 32GB에서 다계통 모델 동시 상주가 불가하므로 *병렬을 포기하고 순차*로 간다.
- **cheap-first 솎기·차등 깊이·통과율 측정을 두지 않는다.** 모든 후보는 모든 관할 심사관을 거친다. 어차피 심사관을 한 번씩 다 로드해야 하므로, 미리 솎아 비용을 줄이는 것은 *재활용 재료를 버리는 손해*일 뿐 이득이 아니다.
- **귀결:** 처리 속도는 느리다. 이는 결함이 아니라 §0-4 핵심 가치의 직접 결과다.

**(2) 구성원 명단 — 계통 분산 7명 + 외부 재심 1명** ⚠️[모델 라인업·성능 = 🟡 T2 다중 교차/신뢰도 ~75%, 역할 배치 = 추정(워크로드 미실측)]

선정 원칙: **정(생성)과 심사관은 반드시 다른 계통**(생성기는 자기 맹점을 못 잡음, §3·§6). 종속핵심(논리·정확성, §9)을 한 계통에만 맡기지 않는다.

## 8. 측면 관할 정의

```
코드 심사관 : 실행·정확성·복잡도. "도는가, 빠른가"
논리 심사관 : 형식 타당성 + 전제 건전성 + 맥락 적합성.
              ※ 'valid'만 보면 §3의 '맥락에 안 맞는 토대론'을 또 놓침 → 건전성·맥락 명문 포함 필수.
대응 심사관 : (보류) 코드↔논리 전제 일치. 미끼 입증 시에만 배치.
```

## 9. 도메인별 측면 독립/종속 — 합산 방식이 갈림

```
코드      : 정확성 ⊥ 구조      (독립 → 가산)
수학풀이   : 정답 ⊥ 풀이과정     (독립 → 가산)
수학추론   : 논리 = 정확성       (종속 → 게이트)
철학      : 논리 = 건전성       (종속 → 게이트)
이론물리   : 수학정합 ⊥ 예측력    (독립 → 가산)
```

- **독립:** 한 면이 0이어도 다른 면 살아남음.
- **종속:** 종속핵심(논리)이 0이면 다른 면 점수가 의미를 잃음 → 게이트로 차단.

## 10. 게이트 + 사전식 래칫 — 절대/상대 레이어 분리 (통합 결정 §0-1)

**핵심:** 두 질문을 두 장치가 나눠 답한다.

### 10-1. 1단계 — 게이트 (통과/탈락, 절대 컷, 곱셈식)
- 종속핵심(§9)이 무너졌나? 무너지면 즉시 탈락, 줄세우기 진입 차단. 게이트 없이 우선순위만 쓰면 "논리 빵점인데 코드 100점이라 1등" 좀비 후보가 샌다.
- **이 단계가 [B]의 절대 점수·Hard Gate·단계별 threshold가 사는 자리다.** "통과 자격" 판정에는 절대 눈금을 쓴다(§17~20).

### 10-2. 2단계 — 사전식 래칫 (순위, 상대 변화)
게이트 통과분끼리.
- **사전식 비교:** 우선순위는 *더하지 않는다.* 최중요 축부터 비교, 갈리는 순간 거기서 결정, 하위 축은 안 봄. **덧셈이 아니라 순서대로 거르기** → "덜 중요한 면이 더 중요한 면을 상쇄"가 구조적으로 불가능.
- **상대점수 기준:** 절대점수 아님. *이전 챔피언 대비 유의미한 변화가 있나*로 각 축 비교. → "동률" 개념이 성립 안 함(비교 대상이 점수가 아니라 변화 폭). v0.9의 *동률밴드(ε)는 폐기* — 절대 눈금 가정이라 불필요.
- **측면별 챔피언 누적(래칫):** 한 후보가 모든 면에서 최고일 필요 없음. 면마다 챔피언이 다르면 §11 라이브러리에 각각 보관했다가 합성.
- **우선순위 줄 출처:** 줄은 *목적별로 뒤집힌다*(프로토타입 vs 라이브러리). 시스템이 추론하지 않고 **과제 입력 시 사용자가 명시.** 점수 *생산*은 심사관 자가선별, 줄 *선택*은 과제 정의의 일부.

> **용어:** **사전식(lexicographic)** = 우선순위 높은 축부터, 갈리면 거기서 끝, 하위는 무시. **래칫(ratchet)** = 한 방향 톱니, 뒤로 안 돌아감(챔피언은 갱신만, 후퇴 없음).

> **레이어 분리 한 줄 요약:** *절대 점수로 "들여보낼지"를 정하고, 상대 변화로 "누가 챔피언인지"를 정한다. 절대 눈금이 순위까지 침범하지 않으므로 [A]의 상대화 철학이 보존되고, [B]의 구체 기준표도 통과 컷으로 그대로 살아남는다.*

## 11. 지연 재사용 라이브러리 + 목적 주석

- "코드는 틀렸지만 논리구조는 옳다"가 나오면, 옳은 논리를 *지금 후보에 억지로 붙이지 않고*(즉시 결합 시 코드 바뀌면 깨짐) **독립 부품으로 라이브러리 보관** → 코드까지 맞는 미래 후보가 가져다 씀. 결합을 시간축으로 푼다.
- **합성물 사전 보증 안 함:** 라이브러리에서 따와 조립한 산출물은 일반 후보처럼 필터에 던진다. 틀리면 불합격→재분해.
- **보관 시 동반 메타(벌거벗은 보관 금지):**
  ```
  - 성립 전제   (참 조건 — 정적)
  - 검증 맥락   (어디서 통과했나)
  - 재사용 조건 (works_when / fails_when)
  - 목적 주석   (무엇을 위해 썼나 — why/의도)
  - 검증 상태   (preserved_verified vs salvaged_unverified)
  ```
- **목적 주석의 의미:** 전제(정적 조건)와 다른 차원. 목적까지 적히면 *기능적 유추*가 가능 — 구조가 달라도 *목적이 같으면* 옮겨붙음(박쥐 날개↔새 날개식 수렴진화). 표면 유추(구조 베끼기)와 구조적 유추(기능 옮기기)를 가르는 선.
- **재사용 2단 매칭:** ①목적 매칭(새 문제가 같은 why?) → ②전제 매칭(조건 참?). 목적 안 맞으면 전제 따지지 않고 스킵.
- **목적 주석도 검증 대상:** LLM의 post-hoc 날조(우연히 생성하고 사후에 그럴듯한 목적 지어냄) 차단. **정이 주장 → 반이 승인**해야 박힘.

## 12. 부정예시 되먹임

- 필터 누설 사례를 생성기에 *부정 예시*로 먹여 입구 궤변 절대량 감축. 검증(출구)과 prior 조정(입구)은 *곱하는 관계*.
- **점화:** 필터 calibration 통과 후에만(미보정 오판을 prior로 굳히면 멀쩡한 방향 조용히 죽음). 그 전엔 기록만(shadow).
- **"학습" 정의:** 파인튜닝 아님(32GB 비현실). 부정예시를 검색해 *생성 프롬프트에 문맥 주입.*
- **가시성:** 회수 조각과 동일(S3→전 단계).
- ※ 이 §12가 [C] 궤변생성기 4단계 루프의 "[4단계] 생성기 입력 문맥 주입"과 동일 메커니즘이다(§28-4).

## 13. 메타검증 — 검사관을 검사하는 장치

세 장치를 한 집으로 묶는다.

- **미끼 프로브(known-flaw probe):** 알려진 결함 심어 블라인드 투입. 측면 발견율(A) 실측 + 게이트 필요성 + 이음새 심사관 필요성 공용 검증.
- **궤변 생성기(adversary):** *현재 필터를 통과하는 가장 그럴듯한 궤변*을 만드는 게 임무. 통과한 궤변 = 필터 구멍 → 패치. OR 게이트에선 필터가 외통수라 거의 필수. → **[C] Part IV에서 독립 구동체로 상세화.**
- **블라인드 재투입:** 통과 산출물 일부를 재심 반에 블라인드 재투입. 뒤집히면 *누설 후보* 1건.
- **누설 확정:** 재심 뒤집힘은 *후보*. 확정 = 도구 증거 또는 사람 확인.
- **재심 실행(v1.3):** 위 블라인드 재투입과 모든 재심은 **자동 LLM 호출이 아니라 재심 요청 문서 출력**으로 처리한다(§0-5). 시스템은 §13-1 규격의 문서를 떨구고, 사용자가 외부 Claude(8번 구성원)에게 전달해 받은 판정을 재입력한다.

### 13-1. 재심 요청 문서 규격 (Appeal Request Packet)

**목적:** 시스템 내부 LLM이 못 잡았을 수 있는 결함을, *뿌리가 다른 외부 계통(Claude)* 이 단서 없이 재검(발견 A)하도록 한다. Claude API 키 없이 구독 채팅만으로 이종 검증을 얻는 경로.

**언제 문서를 출력하나 (트리거 — 하나라도 해당 시):**
1. **블라인드 재투입 뒤집힘** — 통과 산출물이 재투입에서 판정이 바뀜(누설 후보).
2. **빈 측면 탐지** — 아무 심사관도 안 집은 측면 발생(§7). 무심사 통과 위험.
3. **종속핵심 게이트 경계** — 논리·정확성(§9) pass/fail이 애매한 케이스.
4. **`needs_human_review` 판정** — Gate 3에서 AI 단독 확정 곤란(§19-3).
5. **측면 심사관 충돌** — 한 심사관 pass, 다른 심사관 reject로 갈림.

**문서에 반드시 담기는 것 (Claude가 단서 없이 독립 재심하려면 필요한 최소 정보):**
```
[1] 메타       : 후보 ID · 도메인 · 우선순위 줄(§10-2) · 재심 트리거(위 1~5 중)
[2] 원 과제    : 목표 1문장(S1) + 사용자가 준 핵심 조건
[3] 후보 원문  : 검증 대상 산출물 전체(코드/증명/논증 그대로)
[4] 1차 심사   : 각 측면 심사관의 판정(reject/revise/pass) + 점수 + 핵심 사유
[5] 하드체크   : 있으면 그대로 — 컴파일/테스트 러너 결과, SymPy 동치성, 차원검산
[6] 의심 지점  : 왜 재심에 올라왔나(어느 면이 갈렸나 / 어느 측면이 비었나)
[7] 재심 질문  : Claude에게 묻는 것을 구체적으로. 예) "이 증명이 결론을 전제로
                재사용(선결문제 요구)하지 않는가?" "테스트가 하드코딩으로 통과된 건 아닌가?"
[8] 요구 출력  : Claude가 따라야 할 형식(아래 재입력 스키마) — 시스템 재투입을 위해 고정
```

**Claude 재심 결과 재입력 스키마 (사용자가 Claude 답변을 이 형태로 시스템에 입력):**
```yaml
appeal_result:
  candidate_id: "T-001"
  reviewer: "external_claude"        # 계통 = Anthropic(이종)
  verdict: "uphold | overturn | uncertain"   # 1차 판정 유지 / 뒤집기 / 불확정
  defect_found:                       # overturn 시 구체 결함
    - type: "선결문제요구 | 가짜해 | 하드코딩 | 차원불일치 | 빈측면 | 기타"
      where: "후보 내 위치"
      why: "왜 결함인가(1차가 놓친 이유 포함)"
  leak_confirmed: true | false        # 누설(false positive) 확정 여부
  salvageable_part: "폐기 시 살릴 부품(있으면)"   # → §11 라이브러리 환원
  feedback_to_thesis: "다음 정이 피할 생성 규칙"  # → §12·§28-4 부정예시 되먹임
```

**재입력 후 처리:**
- `overturn + leak_confirmed:true` → 누설 1건 확정 → §28-3 필터 패치 + §28-4 생성기 부정예시 주입.
- `salvageable_part` → §11 라이브러리(목적·전제 메타 동반) 보관.
- `uncertain` → 사용자가 영점으로 최종 판단(§3 사람 검증). 강제 결론 금지.
- ⚠️ **주의:** 외부 Claude도 *재심(2차)일 뿐 절대 정답이 아니다.* 1차와 충돌 시 "어느 쪽이 옳다"가 아니라 *충돌 자체가 누설 후보 신호.* 최종 확정은 도구 증거 또는 사람(§13 누설 확정 원칙 불변).

> **복붙용 템플릿(그대로 채워 Claude 채팅에 붙여넣기)은 부록 C 참조.**

## 14. 종료 = 라이브러리 환원 / 메타게임 적응

- **종료 판정:** 사전식 래칫이 멈춤(어느 축도 이전 챔피언 대비 유의미 개선 없음).
- **종료물 = 현재 메타 기준 *잠정* 챔피언**(영구 정답 아님). 상태는 바뀔 수 있다.
- 종료물도 측면별 분해 → 라이브러리 보관(§11 메타 동반).
- **메타게임 적응:** 메타(판을 지배하는 기준)가 변하면 도태된 부품은 `deprecated`(잠듦, 삭제 아님), 유리해지면 재소환. 못 쓰게 된 최선 = 손실이 아니라 *도태 정보*(진화의 멸종처럼).
- **재소환 트리거:** ①1순위 사람 통보(언어 신버전 깨짐·연구 뒤집힘). ②프로그램화 시 *문헌 발행률 변화점 탐지(change-point detection)* — 메타가 바뀌면 그 위 논문이 쏟아지므로 평상시 대비 *급증*을 잡음(절대 기준선 불필요, 상대 급변만).

> **용어:** **메타게임** = 지금 판을 지배하는 전략 환경. 바뀌면 어제의 1티어가 오늘 쓰레기, 단 보관돼 있다 다시 유리해지면 부활. **변화점 탐지** = 절대 수치가 아니라 흐름이 꺾이는 지점을 잡는 통계 기법(추세 전환점).

## 15. 성공 = 게이트 + 쓸모

- 도메인별 성공 기준표를 *따로 만들지 않는다.* 성공은 이미 두 부품으로 정의됨:
  ```
  종속핵심(논리·정확성) : 검증 = 필요조건. 게이트로 차단(§10).
  그 외 측면          : 쓸모 = 충분조건. 검증 없이 통과 가능.
                       검증은 사후에 와서 리팩토링(범위 재조정).
  ```
- **유용성 기준(use-conditional) 성공:** 진리 기준 아님. "쓸모"의 판정도 사후 — 사용자가 써보고 도움 되면 입증, 안 되면 도태.

## 16. 합의 결정 (정의·일관성)

1. 되먹임은 필터 calibration 후 점화.
2. 누설 확정은 사람/도구 앵커, LLM은 눈금만.
3. 부정예시 가시성 = 회수 조각과 동일(S3→전 단계).
4. "학습" = 파인튜닝 아니라 프롬프트 문맥 주입.

# Part III — 반 필터 검증 기준 구현 규격 (출처 [B])

> **위치 확인:** 이 Part 전체는 §10-1 **게이트(절대 컷)** 안에서 작동하는 구체 기준표다. 여기의 점수·threshold는 "통과 자격" 판정용이며, "순위"는 §10-2 상대 래칫이 따로 맡는다.

## 17. 판정 체계

### 17-1. 판정값
| 판정 | 의미 | 다음 이동 |
|---|---|---|
| `reject` | 핵심 기준 미달. 폐기 | 실패 메모리 저장 |
| `revise` | 잠재력은 있으나 보완 필요 | 이전 또는 같은 단계 정으로 회송 |
| `pass_to_next_gate` | 현재 단계 통과 | 다음 단계 정/반 듀오로 이동 |
| `accepted_synthesis` | 최종 승급 통과 | 사용자에게 제출 + 1차 정에게 되먹임 |
| `needs_human_review` | AI 판정만으로 확정 곤란 | 사용자 검토 필요 (§7 빈 측면 탐지와 동일 신호) |

### 17-2. 점수 해석 (게이트 통과 컷 — 절대 눈금, §10-1 한정)
| 점수 | 해석 | 처리 원칙 |
|---:|---|---|
| 0-39 | 실패 | reject |
| 40-59 | 약함 | 대부분 revise 또는 reject |
| 60-74 | 최소 통과권 | 1차/2차 통과 가능, 3차는 부족 |
| 75-89 | 강함 | 핵심 후보로 승급 가능 |
| 90-100 | 매우 강함 | 최종 후보급 |

> ⚠️ **레이어 주의:** 위 절대 점수는 *통과/탈락 컷*으로만 쓴다. 통과한 후보들 사이의 우열은 이 점수로 줄세우지 **않고** §10-2 상대 래칫으로 정한다. (절대 눈금이 순위까지 침범하지 않게 분리.)

### 17-3. Hard Gate와 Quality Metric
- **Hard Gate:** 하나라도 치명적으로 실패하면 점수와 무관하게 reject 또는 revise로 내려간다.
- **Quality Metric:** 후보의 강도를 0-100으로 평가한다. 단계별 threshold 이상이어야 통과한다.

## 19. 3단계 게이트 기준 (결합 구조: 시간축 = 3단계, 각 단계 내부 = 다측면 자가선별)

### 19-1. Gate 1 — 생존 게이트
목적: 완전 노이즈 제거. 잠재력 있는 씨앗은 살린다.
통과 조건: 목표 관련 / 문제 해결 방향 / 기존과 약간이라도 다름 / 검증·비판 가능 형태로 발전 가능 / 완전 말장난·포장만 아님.
판정: `pass_to_next_gate`(2차로) · `revise`(1차 정 회송) · `reject`(실패 메모리).

### 19-2. Gate 2 — 정제 게이트
목적: 실험/논증/구현 가능한 구조로 다듬기.
통과 조건: 핵심 주장 한 문장 정리 / 주요 용어 정의 / 전제-결론 연결 가시 / 예상 반론·실패 조건 2개 이상 / 검증 방법 구체화 / 기존 대비 개선점 / 다음 루프 개선 지점.
판정: `pass_to_next_gate`(3차로) · `revise`(2차 정) · `reject`.

### 19-3. Gate 3 — 학술 승급 게이트
목적: 사용자 제출 최종 합 선별. 학계 검토 후보 수준.
통과 조건: 논리적 자기모순 없음 / 핵심 개념 정의 안정 / 주장 범위 명확 / 기존 논의 대비 위치 설명 / 실질 차별점 / 주요 반례·반론 대응 / 숨은 전제 명시 / 실패 조건·적용 한계 명시 / 검증·반증 가능 / 후속 연구·구현 질문 생성 / 과장·만능 주장 포장 없음.
판정: `accepted_synthesis`(사용자+1차 정 되먹임) · `needs_human_review`(사용자 검토) · `reject`.

# Part IV — 적대적 진화 서브시스템: 궤변생성기 (출처 [C])

> **위치 확인:** §13 메타검증의 `궤변 생성기(adversary)`를 독립 구동체로 확장한 것. GAN(생성적 적대 신경망)의 "위조지폐범 vs 경찰" 구도를 차용해 반 필터의 맹점을 강제 연수한다.

## 26. 핵심 구동 — 위조지폐범과 경찰 (GAN 효과)

궤변생성기는 스스로 '참/거짓'을 완전 구별해서 작동하는 게 아니다. **역할(Role) 비대칭성 + 심사관 규칙의 노출**을 이용해 필터의 한계선(Boundary)을 난타하는 화이트 해커(침투 테스터) 역할을 한다.

```
[심사관 락업 루브릭 노출] ➔ [궤변생성기의 허점 들이받기] ➔ [최정예 쓰레기 양산]
        ↓
[정(생성기) 입구 프롬프트에 부정예시(Block-rule) 주입] 🤼 [사람/도구의 누설 확정 및 심사관 패치]
```

## 27. 역방향 프롬프팅 (Inverse Prompting)
궤변생성기에게 심사관(반)의 도메인별 락업 루브릭(하드코딩 즉시 탈락·순환논증 차단·가짜해 필터 등 = Part III)을 컨텍스트로 전면 노출한다.
- **지시문 앵커:** *"너는 현재 배치된 심사관 필터를 무력화하는 논리 엑스레이 장비다. 심사관 규칙을 형식적으로 100% 만족해 Pass 도장을 받아내되, 실제로는 치명적 논리 결함·꼼수가 숨은 '최상급 궤변/우회 코드'를 제조하라."*
- **공통 뿌리의 역용:** 궤변생성기는 심사관과 모델 뿌리를 공유하므로, 심사관 LLM이 어떤 문장 구조·대수적 사각지대·톤앤매너에 잘 속는지 가장 잘 알고 이를 정밀 타격한다.

## 28. 4단계 적대적 진화 루프

### 28-1. [1단계] 최정예 불량품 가속 생성
궤변생성기가 락업 필터를 뚫으려 발악하며, 인간도 쉽게 낚일 고난도 결함물(겉만 번지르르한 쓰레기)을 시스템 내부에서 대량 생산.

### 28-2. [2단계] 인간/도구 앵커를 통한 '누설 확정'
인간 검증원(사용자)은 수천 개 일반 후보를 모니터링하지 않는다. **필터를 기계적으로 뚫고 올라온 궤변생성기의 최정예 산출물만 샘플링 검수.** 명백한 개소리·꼼수 코드가 필터를 통과해 있으면 그 순간 **'필터 누설(False Positive)' 확정.**

### 28-3. [3단계] 심사관(반) 필터 취약점 패치
누설 궤변의 구조·우회 패턴을 분석해 심사관(반)의 관할 정의·페르소나 지시문에 명문 규칙으로 즉시 추가(패치).
- *예(코딩):* "단위 테스트 결과 객체를 가짜로 생성해 return하는 변칙 패치 패턴을 즉시 Reject 목록에 추가."
- *예(철학):* "결론의 핵심 명제를 전제에 단어만 바꿔 재진술한 선결문제 요구 오류 패치 패턴 추가."

### 28-4. [4단계] 생성기(정) 입력 문맥 주입 (거울 치료)
뚫렸던 고난도 궤변 구조·실패 데이터는 절대 버리지 않는다. **생성기(정)의 상위 프롬프트 문맥(In-context)에 최악의 부정예시(Block-rule)로 직접 주입.** 정은 입구 단계부터 *"이런 고난도 말장난·우회 코드·논리 도약은 출고 즉시 사후 도태된다"*를 인지하게 되어, 입구 후보 품질 자체가 진화.
> ※ 이 4단계가 §12 부정예시 되먹임과 동일 메커니즘.

## 29. 도메인별 적대적 공격/방어 축

### 29-1. 코딩 (보상 최적화 공격)
- **공격:** 테스트 통과만 위한 특정 케이스 하드코딩 / 빈 `try/except`로 런타임 에러 삼키기 / 린트 회피용 무의미 더미 코드.
- **방어:** 보상 최적화 즉시 탈락 조건(§20-3 Reject-if) 가동 + 하드체크(컴파일러/테스트러너) 연동 패치.

### 29-2. 수학·이론물리 (경계 조건·양화사 공격)
- **공격:** 분모 0·정의역 경계 슬쩍 누락해 가짜해(Extraneous Solution)를 정답으로 위장 / 특수 상황(무마찰계) 결과로 일반 물리 법칙 유추 증명.
- **방어:** SymPy 기반 대수적 동치성 하드체크 + 양화사(모든/존재) 충돌 검산 기계 필터링(§20-1·20-2·20-4).

### 29-3. 자연어 철학 (개념 도약·순환논증 공격)
- **공격:** 전제-결론 사이에 직관적으로 그럴듯한 단어(가치관) 배치해 논리 도약 은폐 / 겉보기 완벽 인과관계지만 시작=끝인 순환논증.
- **방어:** 개념 징검다리 검증 루브릭 타이트닝(§20-6) + 외부세계(사용자) 피드백 기반 사후 도태 유용성 앵커.

## 30. 설계적 확신 — 나도 속고 필터도 속았을 때의 방어선

궤변생성기가 인간 영점(사용자)마저 넘는 신의 한 수급 궤변을 짜내 외부로 출고되더라도 구조적 파멸은 없다.
1. **외부 세계의 도태 압력:** 내부에서 참으로 보였어도 실제 세상에 적용해 모순·무용하면 **쓸모(Utility) 원칙에 의해 무조건 폐기**(§15 유용성 기준).
2. **사후 수거 및 영구 차단:** 외부에서 뺨 맞고 돌아온 고난도 궤변은 시스템의 가장 가치 있는 '진화적 재(Data)'. 즉시 3단계(필터 패치)·4단계(생성기 문맥 주입)로 환원되어 해당 오류 부류에 영구 면역 획득.

# Part V — Kill-switch / 반증조건

## 31. Kill-switches (확정 반증조건, 출처 [A])
- **[KS-반/루프]** 풍부 피드백을 줘도 루프가 예산매칭으로 best-of-N을 못 이기면 → 루프 폐기, 반+best-of-N만. 이기면 → 정당화, "피드백 풍부함"이 핵심 노브.
- **[KS-LLM반]** 미끼에서 다계통 LLM 반이 비검증 도메인 결함을 높은 비율로 잡으면 → 단독 충분. 낮으면 → 접지 필수.
- **[KS-이음새]** 진짜 이음새 결함을 측면 심사관단이 잡으면 → 이음새 심사관 영구 불필요. 빠져나가면 → 그때 배치.
- **[KS-게이트]** 종속 도메인에서 게이트 없이 좀비 후보가 한 번이라도 통과 → 게이트 필수. 안 나오면 → 제거.
- **[KS-사전식]** 상대점수가 깔끔한 동률로 자주 떨어져 근소차가 드물면 → 사전식만으로 충분. 근소차 흔하고 좋은 하위축이 자주 버려지면 → 상대화 부족, 기준점 재설계.
- **[KS-라이브러리]** 목적·전제 미기입 부품을 붙여도 충돌 거의 없으면 → 기입 의무 과잉. 충돌 시작되면 → 필수. (부품 20~30개 + 교차 재사용 시점 판가름.)
- **[KS-목적주석]** 목적 주석 무검증 박아도 오매칭 거의 없으면 → 반 승인 과잉. 가짜 목적이 엉뚱 유추 유도하면 → 검증 필수.
- **[KS-빈측면]** 무심사 누설 측면이 실전에서 거의 없으면 → 빈 측면 탐지 과잉. 한 번이라도 결함 실려 출고되면 → 필수.
- **[KS-되먹임]** 미보정 상태로 켜도 초기 오판이 입구 안 죽이면 → shadow 게이팅 과잉. 죽이면 → 필수.

# 부록 A — 용어집
- **OR 게이트:** 통과 1개면 이득인 필터 구조. 입구 품질 측정 안 함.
- **false positive(거짓 통과):** 떨어졌어야 할 결함물의 출고. OR 게이트 최대 위험.
- **best-of-N / 루프:** 독립 N개 중 최고 선택 / 1개를 반 피드백으로 반복 수리.
- **발견(A) vs 수리(B):** 단서 없이 결함 첫 적발 / 짚어주면 고침. 반에 필요한 건 A.
- **사전식(lexicographic):** 우선순위 높은 축부터, 갈리면 끝, 하위 무시.
- **래칫(ratchet):** 한 방향 톱니, 후퇴 없는 챔피언 갱신.
- **메타게임:** 지금 판을 지배하는 전략 환경. 바뀌면 1티어가 쓰레기, 보관됐다 부활 가능.
- **변화점 탐지(change-point detection):** 절대 수치가 아니라 추세 전환점을 잡는 통계 기법.
- **목적 주석:** 부품이 "무엇을 위해" 쓰였나(why). 구조가 달라도 목적이 같으면 옮겨붙는 기능적 유추를 가능케 함.
- **Hard Gate / Quality Metric:** 치명 실패 시 점수 무관 강등하는 통과/탈락 기준 / 0-100 강도 평가.
- **궤변생성기(adversary):** 현 필터를 통과하는 가장 그럴듯한 궤변을 만들어 필터 구멍을 노출시키는 침투 테스터.
- **역방향 프롬프팅:** 심사 기준을 적에게 노출해 "형식상 통과하되 실질 결함" 산출을 유도.

# 부록 C — 재심 요청 복붙 템플릿 (외부 Claude 채팅용)

아래를 그대로 복사해 빈칸을 채운 뒤 Claude 채팅에 붙여넣는다. 시스템이 §13-1 트리거를 만나면 이 형식으로 문서를 떨군다.

```text
[재심 요청 — 다측면 심사 하네스]

너는 이 후보의 외부 재심관(2차 독립 검증)이다. 너는 이 시스템 내부 모델들과
계통(학습 뿌리)이 다르다. 내부 심사관들이 "함께 놓쳤을" 결함을 단서 없이 찾는 것이
네 임무다. 친절한 총평이 아니라, 1차 판정을 유지(uphold)/뒤집기(overturn)/불확정
(uncertain) 중 하나로 판정하라. 너의 판정도 정답이 아니라 2차 의견이며, 1차와 충돌하면
그 충돌 자체가 누설 신호다.

[1] 후보 ID / 도메인 / 우선순위 줄 / 재심 트리거:
    (예: T-001 / 수학증명 / [논리>정확성>형식화] / 측면 심사관 충돌)
[2] 원 과제 목표(1문장) + 핵심 조건:
    ...
[3] 후보 원문(전체, 그대로):
    ...
[4] 1차 심사 결과(각 측면: 판정 + 점수 + 사유):
    ...
[5] 하드체크 결과(컴파일/테스트/SymPy/차원검산 — 있으면):
    ...
[6] 의심 지점(어느 면이 갈렸나 / 어느 측면이 비었나):
    ...
[7] 내가 너에게 묻는 것(구체적으로):
    ...

[요구 출력 — 아래 YAML 형식으로만 답하라]
appeal_result:
  candidate_id: "..."
  reviewer: "external_claude"
  verdict: "uphold | overturn | uncertain"
  defect_found:
    - type: "..."
      where: "..."
      why: "..."
  leak_confirmed: true | false
  salvageable_part: "..."
  feedback_to_thesis: "..."
```

**사용 흐름:** 시스템이 위 문서 출력 → 사용자가 Claude 채팅에 붙여넣음 → Claude가 YAML로 답
→ 사용자가 그 YAML을 시스템에 재입력 → §13-1 "재입력 후 처리"대로 환원/패치.
